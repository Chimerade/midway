Battle of Midway (3–7 June 1942)

Data construction methodology and framework for the future simulation program

Progress status (v2, 12 June 2026). Phases 0 through 7 are framed and tooled: operational SQLite database (79 events, 59 air tracks, 15 instrumented decisions, 61 claims), interactive replay map (carte_midway.html), automated consistency audit, constraint-based position inference engine, and F1 engine (replay of 4 June through the constraint model — 13 checks, 0 inconsistencies, see rapport_f1.md). Still open: systematic source review (17/61 claims verified by 2 sources), georeferencing of reference maps, CAP and submarines, and the F2/F3 engines. The tout_regenerer.sh pipeline rebuilds everything; the map's #hash buffer guarantees synchronization with the database.

1. Guiding principles

The intended program has three functions, each imposing a distinct requirement on the data :

FunctionData requirementMethodological consequence
F1 — Historical replay on a scaled 2D mapContinuous, dated, sourced chronology and positionsEvery position/time must carry a confidence index and a method (observed vs. reconstructed)
F2 — Optimal decision simulationComplete physical constraint model (flight decks, rearming, fuel, delays)Constraints must be parameterized (durations, capacities), not merely described
F3 — Targeted counterfactual (alter one decision, observe consequences)Decisions isolated as entities, with the information available to the decision-maker at the timeStrictly separate actual state of the world from each side's perceived state (fog of war)

Three principles follow :

P1 — Separate facts, reconstructions, and model parameters. The database distinguishes what is attested by a primary source, what is interpolated, and what is a simulation parameter (e.g., the rearming duration for a squadron). The three are never mixed.

P2 — Every datum is a sourced claim, not a truth. Sources contradict one another (times, positions, strengths). The schema stores competing claims (claims) alongside a retained value with justification — making it possible to revise without losing the record.

P3 — Midway is a scheduling problem under incomplete information. The heart of the model is not the combat itself but the flight-deck cycle: launch / recover / spot / rearm / refuel, with the flight deck as the single critical resource per aircraft carrier. The chosen granularity is the squadron (squadron / chūtai), with the deck cycle modeled per carrier.

2. Fundamental conventions (to be frozen before any data entry)

2.1 Time — Midway's first pitfall

Sources use three different time references. Japanese documents (Nagumo's report, deck logs) are in Tokyo time (GMT+9). American summary documents use local Midway time (GMT−12 zone). The gap is 21 hours, spanning a date change: "01:30 on 5 June" in Japanese = "04:30 on 4 June" Midway time. Many errors in the literature stem from this.
Confirmed during review (Phase 8) : the US carrier deck logs use a fourth reference, GMT−10 (ship's time = Midway +2h): the attack on Hiryū is logged there as "1905" for an actual time of 17:05. The time_reference field of each source is therefore mandatory, and every time is entered with its original value in claims.original_value.

Adopted convention : all times stored in the database are in Midway local time (GMT−12), in ISO 8601 format (1942-06-04T10:22:00-12:00), with a time_uncertainty_min field. The source's original time and its time zone are preserved in the claims table. This follows the convention of Parshall & Tully (Shattered Sword), the modern reference.

2.2 Space

ElementConvention
StorageWGS84 latitude/longitude in decimal degrees (sources give degrees-minutes: converted on entry, original value retained)
Map displayAzimuthal equidistant projection centered on Midway (28°12′N, 177°21′W) — exact distances and bearings from the center, negligible distortion over ~700 nm
DistancesNautical miles (nm); speeds in knots; headings in true degrees
PrecisionEach position carries an estimated position_error_nm. 1942 positions rely on dead reckoning: errors of 20–60 nm are common (e.g., the PBY report at 05:34 placed the Kidō Butai ~40 nm too close to Midway, which contributed to the Hornet air group's fruitless search)

2.3 Identifiers

Stable, human-readable identifiers, prefixed by type: SH-AKAGI, SH-CV6 (Enterprise), SQ-VT8, SQ-AKAGI-KANKO (Akagi's B5N squadron), AC-B5N2, EV-0604-1022-A (event), PR-NAGUMO. Midway Atoll is treated as an "unsinkable aircraft carrier": SH-MIDWAY, with its own runway constraints.

3. Methodological phasing

Phase 0 — Scoping (done — decisions above)

Scope: the entire battle, 3–7 June 1942 (approach, strikes of 3 June, decisive day of 4 June, pursuit of Mogami/Mikuma, loss of Yorktown). Air granularity: squadron. Storage: SQLite. Time/space conventions frozen (§2). Exit criterion: this document approved.

Phase 1 — Source corpus and verification protocol

Assemble the library (§4), assign reliability grades, define the conflict-arbitration rule. Exit criterion: sources table populated; ≥ 2 independent sources identified for each data family (OOB, chronology, positions, technical specifications).

Phase 2 — Static reference (what does not change during the battle)

Complete orders of battle for both sides, ship characteristics, physical constraints of carriers and Midway, aircraft type sheets, squadron strengths as of 04 June at 04:00, chain of command. Exit criterion: tables ships, carrier_constraints, aircraft_types, squadrons, persons complete, each critical value verified by 2 sources.

Phase 3 — Event chronology

Entry of all dated events: detections, messages, orders, launches, recoveries, attacks, hits, damage, scuttlings. Each event = time (converted to GMT−12) + uncertainty + actors + sources. Begin with the ~60 major anchors (§7), then densify. Exit criterion: continuous chronology with no gap > 30 min for major units on 4 June; remaining gaps explicitly listed.

Phase 4 — Tracks and positions

Entry of attested positions (contact reports, deck logs, launch/recovery points), then reconstruction of tracks by constrained interpolation (max speed, known ordered headings). Air raid tracks reconstructed from heading/speed/departure-arrival times. Exit criterion: continuous track (1 point/10 min in interpolated mode) for each naval formation and each raid, replayable on the map.

Phase 5 — Process parameterization (the heart of F2)

Quantify operations: spot/launch/recovery durations per deck, elevator cycles, rearming durations by configuration (torpedo↔bomb), refueling, CAP rotation, transmission and decision delays. Sources: action reports, Shattered Sword (appendices on Japanese deck operations), doctrine. Exit criterion: process_templates / process_steps tables populated, each duration justified.

Phase 6 — Information state (fog of war)

For each major decision: reconstruct what the decision-maker knew, when, and through which channel (the knowledge_states table, fed by messages). Essential for F3: a decision can only be judged against the information that was actually available. Exit criterion: the ~15 major decisions (§7.2) documented with their informational context.

Phase 7 — Replay validation, then simulation engine

Replay F1 and verify that reconstructed events comply with the model's physics (a raid does not fly faster than its slowest aircraft; a flight deck does not launch while recovering). Any violation signals a data or model error to be arbitrated. Only then: simulation engine (§11). Exit criterion: complete replay with no constraint violation; residual discrepancies documented.

4. Source corpus and verification protocol

4.1 Reliability grades

GradeDefinitionExamples
AOfficial contemporary primary sourceAction reports of US ships and task forces; Nagumo's official report (OPNAV translation "Japanese Story of the Battle of Midway"); deck logs; archived radio messages; CINCPAC report serial 01849
BModern scholarly study cross-referencing primary sourcesParshall & Tully Shattered Sword (2005, the reference for the Japanese side); Lundstrom The First Team (US fighter operations, minute by minute); Cressman et al. CClassic secondary source or reputable online databaseMorison; combinedfleet.com (TROMs of Japanese ships); midway1942.org / Battle of Midway Roundtable; Naval History & Heritage Command
DLate testimony or source with known reliability problemsFuchida Midway (1955): account widely refuted on key points (see 4.3); uncorroborated late memoirs; Wikipedia (usable as an index, never as a final source)

4.2 Verification protocol

Rule: every datum entering the simulation (time, position, strength, duration, characteristic) is entered as a claim linked to its source, its grade, and its original value (time in the source's time zone, position in degrees-minutes). The retained value is chosen by: (1) highest grade; (2) in the event of conflict between equal grades, physical consistency with neighboring events; (3) arbitration documented in resolution_note. A status field (verified / single_source / conflicting / estimated / gap) makes the verification state queryable — at any moment one can see what fraction of the database is solid.

4.3 Known controversies to address explicitly

ControversySignificance for the simulation
Fuchida's "five fatal minutes": the idea that Japanese flight decks were covered with aircraft ready to launch at 10:20 is refuted by Parshall & Tully (the strike aircraft were still in the hangars; the CAP monopolized the decks)Radically changes the initial deck/hangar state at the moment of the hits — a major input to the scheduling model
The Hornet air group's heading ("flight to nowhere"): official 240° vs. ~265° according to several reconstructionsAffects the reconstructed track, the isolated loss of VT-8, and the counterfactual "what happens if the Hornet attacks?"
Exact chronology of Nagumo's rearming orders (07:15 order, 07:45 suspension) and the actual state of progress at each instantThis is the textbook counterfactual for F3; requires the best possible reconstruction of hangar cycles
Delayed launch of Tone's scout plane No. 4 and its actual role (the delay paradoxically placed the aircraft in the right position)Informational pivot: determines the time at which the Japanese detected TF-16/17
PBY contact positions (dead-reckoning errors of ~30–40 nm)To be stored in duplicate: the reported position (what decision-makers believed) and the reconstructed actual position
These cases justify the dual-register approach — real world / perceived world: the contact_reports table stores the reported position; the positions table stores the reconstructed position. The F2 function (optimal decision) must be able to run in both modes: perfect information and historical information.

Movement causality convention (v2). Each waypoint in positions carries a cause_event_id: the event (order, decision, damage) that justifies its outgoing segment. Every course change must therefore be its own waypoint, and the audit (rules M1/N1) rejects waypoints without a cause and logged headings inconsistent with the actual route (±45°). Historical course changes are events; reconstruction adjustments are inferences traced in position_inferences and claims — the two are never mixed. Another structural distinction: ships that were hit leave their formation track and receive their own drift track (point of impact → wreck).

10. Replay validation (before any simulation engine)

The F1 replay serves as the model's test bench: reality is played back through the constraint model, and any impossibility flags an error. Automatable checks: ground speed of raids ≤ cruising speed of the slowest type ±10%; range ≥ distance flown (otherwise: ditchings — which are in fact attested, e.g., returning VF-6 aircraft); a flight deck cannot launch and recover simultaneously; a squadron can only be in one place; receipt times ≥ transmission times; consistency of positions/bearings in mutual contact reports. Completeness indicators: % of claims at verified status, residual chronological gaps, closing error of tracks against wreck positions.

11. Target program architecture (outline)

Discrete-event simulation (DES) is the natural approach: state only evolves at events (end of a rearming step, arrival of a raid, receipt of a message), which maps exactly onto the data model. Layers: (1) SQLite database = historical truth + parameters; (2) state engine (ships, squadrons, deck resources) replayable in "historical" mode (F1, driven by events) or "simulated" mode (F2/F3, driven by decisions); (3) decision module — for F2, flight-deck scheduling is formulated as a job-shop problem with resources (constraint solver, e.g. CP-SAT, or heuristics + Monte Carlo for uncertainty); for F3, one decision in decisions is modified and the engine restarts in simulated mode from that point, with the rest of the world following the model; (4) 2D map rendering (true scale, time slider, uncertainty halos, "as perceived by each side" mode).

Methodological safeguard: F2/F3 require a stochastic model of combat outcomes (hit probabilities, CAP/AA attrition) calibrated against historical 1942 engagements (including Coral Sea); without this, the counterfactual will produce illusory certainties. Plan for distributional outputs (N Monte Carlo runs), never a single result.
